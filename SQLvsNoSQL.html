El Legado de Edgar F. Codd

"En 1970, los datos estaban 'atrapados' en el hardware. Codd liberó la información al separarla de su representación física."

Edgar F. Codd, un científico computacional británico trabajando para IBM, publicó el paper "A Relational Model of Data for Large Shared Data Banks". Su gran innovación fue el concepto de Independencia de Datos: los programas no deberían saber cómo se guardan los datos físicamente para poder consultarlos.

Hitos Clave:

12

Codd definió las 12 reglas de Codd (que en realidad eran 13, numeradas del 0 al 12) para definir qué hace que una base de datos sea realmente relacional.

1981

Recibe el Premio Turing (el "Nobel" de la informática) por sus contribuciones a la gestión de datos.

Consistencia Transaccional

En una disertación es vital explicar los dos modelos de consistencia:

SQL: El Rigor ACID

  • (A)tomicity: Todo o nada. Si falla el paso 9 de 10, se deshace todo.
  • (C)onsistency: Los datos cumplen todas las reglas (FK, Not Null).
  • (I)solation: Las transacciones no se estorban entre sí.
  • (D)urability: Una vez guardado, el dato es inmortal ante fallos de energía.

NoSQL: La Flexibilidad BASE

  • (B)asically (A)vailable: El sistema siempre responde, aunque el dato sea viejo.
  • (S)oft State: El estado del sistema puede cambiar sin entrada de datos.
  • (E)ventual Consistency: Los datos se sincronizarán... tarde o temprano.

El Teorema de CAP

Propuesto por Eric Brewer, dicta que un sistema distribuido solo puede tener 2 de 3:

C
Consistencia

Todos ven el mismo dato al mismo tiempo.

A
Disponibilidad

El sistema siempre responde (aunque sea error).

P
Tolerancia Partición

Funciona aunque falle la red entre nodos.

SQL Clásico = C + A | NoSQL Moderno = A + P o C + P

Dinámicas de Crecimiento

¿Cómo reacciona tu base de datos cuando el tráfico explota?

Escalado Vertical (SQL)

"Comprar un camión más grande". Es más fácil de administrar pero tiene un límite físico (máximo de RAM/CPU en un servidor) y el costo crece exponencialmente.

Escalado Horizontal (NoSQL)

"Contratar una flota de camionetas". Usamos Sharding (particionamiento). No hay límite teórico: puedes agregar miles de nodos baratos. Es la base de Google y Amazon.

El "Chanchito Feliz" de Oracle

Larry Ellison lanzó su primer producto como "Oracle v2". La v1 nunca existió. Sabía que nadie confiaría su dinero a una "v1.0" por miedo a los errores.

La Regla de los 100ms

Amazon descubrió que cada 100ms de latencia reducía las ventas en un 1%. De esa obsesión nació el paper de Dynamo, el abuelo de las NoSQL modernas.

✈️

SEQUEL vs SQL

El nombre original era SEQUEL, pero una empresa de aviación británica ya tenía la marca. IBM quitó las vocales para evitar demandas.

🕵️

El término NoSQL

Fue acuñado en 1998 por Carlo Strozzi para su base de datos. Lo curioso es que su base de datos ERA relacional, solo que no usaba el lenguaje SQL.

📱

El Rey Silencioso: SQLite

Es el motor más distribuido del mundo. Está en cada celular, navegador y auto moderno. No es un servidor, es un archivo en tu memoria.

📊

¿SQL ha muerto?

No. Es el 3er lenguaje más demandado. El futuro es el Poliglotismo de Almacenamiento: usar ambas según la parte de la App.

Resumen para tu Conclusión

ELIGE — SQL — SI...

Necesitas integridad financiera (bancos), reportes complejos o tus datos tienen una estructura que no cambiará en años.

ELIGE — NoSQL — SI...

Necesitas escala masiva, velocidad extrema de lectura (Big Data), o manejas datos muy variados como logs o redes sociales.
